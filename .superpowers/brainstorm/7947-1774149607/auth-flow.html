Clerk + Shopify OAuth — Auth Flow & Data Model

How users move from sign-up to connected dashboard

User Journey

Authentication Flow
1 User visits app.stockpilot.com → Marketing landing page
2 Clicks "Sign Up" → Clerk hosted sign-up (email+password or Google)
3 Email verified → Redirect to /onboarding
4 Onboarding wizard: Store name → Connect Shopify (OAuth) → Configure alerts
5 Shopify OAuth callback → Shop linked to User → Initial inventory sync
6 Redirect to /dashboard — fully connected

Data Model Change

New: Users table + Shop relationship
users (NEW)
id bigint PK
clerk_user_id string UNIQUE
email string
name string
onboarding_completed_at timestamp
created_at / updated_at
1 → N
shops (MODIFIED)
id bigint PK
user_id bigint FK → users
shop_domain string
access_token encrypted
plan, settings, ...
Key change: User owns Shop(s). Multi-tenancy stays on Shop — acts_as_tenant :shop unchanged. A User can connect multiple Shopify stores.

Auth Architecture

Request Flow — Every Authenticated Request
Browser → Clerk JS adds session cookie automatically
↓
Rails → clerk-sdk-ruby middleware validates session
↓
ApplicationController → current_user from Clerk session
↓
ApplicationController → current_shop from user's active shop
↓
ActsAsTenant → scopes all queries to current_shop

What Changes vs What Stays

New / Changed

  • clerk-sdk-ruby gem + Clerk JS CDN
  • New users table + User model
  • shops.user_id foreign key
  • Landing page → marketing + Clerk sign-in
  • Onboarding wizard (3 steps)
  • ApplicationController auth rewrite
  • Replace OmniAuth Shopify init flow

Stays the Same

  • acts_as_tenant :shop — all models
  • Shopify OAuth for store connection
  • All existing controllers/views/jobs
  • Sidekiq, Redis, Sentry config
